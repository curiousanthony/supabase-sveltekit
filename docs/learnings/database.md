# Database / Drizzle / Storage Learnings

Project-specific patterns discovered during development. Read this file before writing schema, migrations, service logic, or Storage operations.

- **DB mutations**: Multi-step updates (replace + link, etc.) belong in `db.transaction()` with optimistic concurrency (`WHERE status = expected`) when races matter.
- **Storage replace/remove**: Replace or remove Storage objects only after a successful upload and DB update; validate object keys against expected prefix (e.g. `workspaceId/`) before `remove()`; set Sharp `limitInputPixels` when processing user images (decompression DoS).
- **Audit logging**: If `logAuditEvent` runs inside a transaction, pass the transaction client and surface errors — avoid fire-and-forget in critical paths.
- **Lifecycle pre-validation**: Pre-validate every transition (status, financing, dates, replace/replace-target) BEFORE `generateDocument` writes Storage objects or DB rows — failed validation downstream leaves orphaned uploads.
- **Server-side preflight parity**: When a compliance/preflight check guards a flow, it must run on EVERY entry point that triggers it (Documents action, Suivi quest sub-action, batch jobs, cron) — covering only the most visible path is a compliance gap.
- **Seed parity across creation paths**: When an entity can be created via multiple paths (e.g. `/formations/creer` AND `deal.closeAndCreate`), seeding side-effects (quests, sub-actions, default rows) MUST be replicated or extracted into a shared `seed{Entity}Defaults(tx, id)` helper from day one — silent divergence produces "ghost-empty" entities that look like UI bugs but are actually missing seed data.
- **Raw DB row shapes in pure services**: Anchor pure service functions on the shapes the DB actually returns (numeric columns come back as strings; use a small `toNumber` helper) rather than on mapped domain models — keeps loaders a one-liner and makes services trivially unit-testable.
- **Dual-layer bounded integer validation**: Bounded integer columns need both a DB `CHECK (value BETWEEN 1 AND N)` AND server-side coercion; DB-only surfaces 23514 errors to the user; server-only lets service-role callers bypass it. Coerce out-of-range → `null` silently (don't throw) so inline-edit UX stays smooth.
- **Migrations**: Hand-edited or hand-ordered SQL can diverge from Supabase's tracking — before applying or relying on `migration up`, confirm what `schema_migrations` actually contains.
- **Scheduled tasks (Vercel)**: Use Vercel Cron Jobs (`vercel.json` `crons` array, `CRON_SECRET` bearer auth, service-role Supabase client) for SvelteKit-on-Vercel scheduled work — cron runs without a user session, so plan auth and RLS bypass explicitly.
- **Drizzle circular schema imports**: Bidirectional `foreignKey` references between two schema files are safe — the `foreignKey(...)` call lives inside the lazy `(table) => [...]` callback, which fires after both modules finish loading. Verify with `bun -e "import('./schema').then(s => console.log(s.tableA.fkColumn))"` before worrying; do NOT "fix" it by merging files — the canonical pattern is one table per file.
- **Idempotent Supabase migrations**: `IF NOT EXISTS` only works directly for `CREATE TABLE`, `ADD COLUMN`, and `CREATE INDEX`. For `CREATE TYPE`, `ADD CONSTRAINT`, and `CREATE TRIGGER`, wrap in `DO $$ BEGIN IF NOT EXISTS (...) THEN ... END IF; END $$` blocks. The project's `updated_at` trigger helper is `public.set_updated_at()` — use it consistently instead of `moddatetime()`.
- **Drizzle schema mocking (tests)**: When unit-testing a pure service that imports a Drizzle table only as a type anchor for `InferSelectModel`, stub it with `vi.mock('$lib/db/schema', () => ({ tableName: Symbol('tableName') }))` — a symbol satisfies the module graph without pulling `postgres-js`/`pglite` into Vitest. Anchor test inputs on raw DB row shapes (numeric columns are strings).
