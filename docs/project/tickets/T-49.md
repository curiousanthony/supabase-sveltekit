Follow-up to T-48 (deal→formation quest seeding). Two debts identified during code review:

1. **Duplication**: ~55 lines of quest + sub-action seeding are now copy-pasted between `src/routes/(app)/formations/creer/+page.server.ts` (lines ~250-308) and `src/routes/(app)/deals/[id]/+page.server.ts` (lines ~312-365). Extract into a shared helper `seedFormationQuests(tx, formationId, userId, type, funding)` in `$lib/formation-quests.ts` (or sibling). As a side benefit, the `creer` flow's known bug (uses `db.insert` outside the transaction wrapper, leaving orphans on failure) becomes fixable in the helper itself.
2. **Tests**: T-48's acceptance criterion called for unit tests covering (a) deal-with-funding seeded set, (b) deal-without-funding seeded set, (c) rollback on quest-insert failure. None were written because the recipe for `bug + small` skips TDD. After extraction, the helper becomes the natural unit-of-test (no need to mock SvelteKit form actions).

## acceptance

- `seedFormationQuests` extracted to `$lib/formation-quests.ts` with signature `(tx, formationId, userId, type, funding) => Promise<void>`
- Both `formations/creer` and `deals/[id]` call it (deals/[id] already inside its existing transaction; creer migrated to wrap modules+quests in one transaction together)
- Unit tests in `src/lib/formation-quests-seed.test.ts` cover: (a) base set seeded when type/funding null, (b) funding-filtered set when funding provided, (c) `blockedByActionId` wired correctly, (d) rollback leaves the parent transaction in a consistent state (mocked tx.insert throwing)
- No behavior change in either call site beyond the creer outside-tx fix
