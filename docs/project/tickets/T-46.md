Carry-over from the T-13 security audit: `formation_audit_log` INSERT on warning-override does not bind `user_id` to `auth.uid()` from a server-trusted source. A user with a stale or substituted JWT could log overrides under another identity. Tighten the insert path so `user_id` is always set from the SvelteKit `event.locals.user.id` (or equivalent) and never from client-supplied input.

**Orchestrator hint**: `type: security` + `estimate: small` → recipe `0 → 4 → 5(security-analyst) → 7b`. No brainstorm — the fix is mechanical. The security-analyst owns the post-fix verification.

## acceptance

- Every `formation_audit_log` insert (documents action, suivi quest action, any other call sites) reads `user_id` only from server-side authenticated context
- No code path accepts `user_id` from `FormData` or query params
- security-analyst sign-off recorded in the ticket log
