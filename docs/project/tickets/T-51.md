`formations.tjm_formateur` is vestigial — `formation_formateurs.tjm` already stores TJM per formateur and the Finances tab cost calculation already uses the per-formateur values exclusively. Keeping the formation-level column creates inconsistency when there are multiple formateurs (which value is canonical?) and clutters the Fiche with an irrelevant input. This ticket migrates the data, drops the column, and removes the corresponding UI control.

**Migration policy (per V1 decision):** for orphan formations (have `tjm_formateur` but no `formation_formateurs` rows), **drop the value silently**. We accept the information loss because the data is unused downstream and creating ghost formateur rows would corrupt the cost calc.

## acceptance

- Pre-migration script: for any formation with non-null `tjm_formateur` AND existing `formation_formateurs` rows where any `tjm IS NULL`, backfill those rows with the formation-level value
- Pre-migration script: for orphan formations (non-null `tjm_formateur` AND zero `formation_formateurs` rows), drop the value silently — no warning, no placeholder row
- Drop column `formations.tjm_formateur`
- Remove `tjmFormateur` from `allowedFields` in `src/routes/(app)/formations/[id]/fiche/+page.server.ts`
- Remove the TJM formateur card from the Fiche (handled in T-54 if not done here)
- Finances `formateursCost` computation unchanged (already uses `formationFormateurs.tjm`)
- Migration is reversible (down migration re-adds the column as nullable)
