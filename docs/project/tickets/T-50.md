T-20 hardened the `formation_documents` RLS policy with an AND across the formation
and deal branches (a row is in scope only if every set FK belongs to the caller's
workspace). The security analyst flagged two regression tests that should exist
to catch any future OR-permissive regression:

1. As user-A, attempt `INSERT` with `(formation_id = formationA.id, deal_id = dealB.id)`
  → expect `42501` (RLS denial). Today the AND policy denies this; if anyone
   weakens it back to OR (or drops the deal-branch WITH CHECK), this test fires.
2. As user-A, attempt `UPDATE` setting `deal_id = dealB.id` on a row they own
  via `formation_id = formationA.id` → expect `42501`. Catches the same
   pivot attack via UPDATE.

The repo has **no RLS test infrastructure today** (no pgTAP install, no Vitest +
service-role-vs-anon-auth harness). The first ticket to add a test must also build
that infrastructure.

Recommended approach (lighter): add a Vitest suite under `src/lib/db/rls/` that:

- Creates two workspaces (A, B) and a user in each via the service role
- Acquires JWTs for each (admin.users API or `signInWithPassword` against a seeded user)
- Uses postgres-js with `Authorization: Bearer <jwt>` to exercise RLS as that user
- Runs the two assertions above, plus baseline coverage:
  - SELECT/UPDATE/DELETE cannot reach a row whose only FK is in another workspace
  - Pre-formation row (formation_id NULL, deal_id set) is correctly scoped via deal
  - Post-conversion row (both set, same workspace) is reachable

Heavier alternative: pgTAP under `supabase/tests/` with `supabase test db`. More
idiomatic for SQL-level RLS tests but requires the pgTAP extension and CI plumbing.

## acceptance

- A test harness exists under either `src/lib/db/rls/*.test.ts` (Vitest) or
`supabase/tests/*.sql` (pgTAP) that runs in `bun run test` (or a dedicated
`bun run test:rls` script wired into CI)
- The two T-20 regression cases above are covered and pass against the current
migration
- Baseline coverage for the four `formation_documents_{select,insert,update,delete}`
policies under formation-only, deal-only, and both-set scenarios
- `docs/decisions/` doc if Vitest vs pgTAP needs justification
