The current `formationType` enum (`Intra`, `Inter`, `CPF`) conflates *delivery format* (Intra vs Inter vs Sur-mesure) with *CPF eligibility* (a property of the certification, not the format). Marie hits this every time she has an Intra formation that's also CPF-eligible — she can't represent it. After the multi-source funding model (T-50/T-52) ships and we have production usage data, untangle this.

**IMPORTANT — quest engine audit required first.** The quest engine currently branches on `type === 'CPF'`. Before refactoring, audit each quest individually to determine whether the branching means:

- "this formation is CPF-eligible" → branch on the new `cpfEligible` boolean
- "this formation is CPF-funded" → branch on `funding_source.type IN (CPF, CPF_Transition)`
- something else (e.g. simplified subset for individual learners)

Different quests may need different fixes. Don't refactor blindly.

## acceptance

- Quest engine audit performed first: for every quest that branches on `type === 'CPF'` (search across `src/lib/formation-quest-*.ts`), document in this ticket's `## log` what the branching semantically means and which replacement applies
- `formation_type` enum updated: drop `'CPF'`, add `'Sur-mesure'`
- New column `formations.cpf_eligible` boolean default false
- Data migration: every formation with `type = 'CPF'` becomes `type = 'Inter'` + `cpf_eligible = true` (or alternative based on the audit)
- Quest engine refactored per the audit findings; existing tests pass
- Fiche Type combobox shows new options (Intra / Inter / Sur-mesure) ; new "CPF-éligible" Switch added next to Référencement toggle on Fiche
- Documentation note added to `docs/decisions/` recording the audit conclusions for future reference
