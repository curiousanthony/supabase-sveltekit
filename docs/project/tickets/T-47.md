Persistent finding across the last several security audits: the Drizzle client we use server-side connects with credentials that bypass RLS. Workspace isolation is enforced by hand in every query (e.g. `eq(formations.workspaceId, workspaceId)`) preceded by `verifyFormationOwnership(...)`. This works today but is brittle: any new query that forgets the workspace clause is a data-leak. Decide the long-term posture: (a) keep Drizzle bypassing RLS and add a lint/codemod that requires a workspace clause, (b) move to a per-request Postgres role + RLS, (c) wrap the Drizzle client behind a workspace-scoped factory that injects the clause automatically.

**Orchestrator hint**: `type: tech-debt` but **NEEDS BRAINSTORM** — this is an architectural decision with non-trivial trade-offs (perf, dev ergonomics, audit story). Recipe should be the **full feature path** `0 → 1 → 1b → 2 → 3 → 4 → 5 → 6 → 7 → 7b`, with `architect` and `security-analyst` co-leading Phase 1, and `decision` doc written in `docs/decisions/` before any code lands. Do **not** start at Phase 4.
