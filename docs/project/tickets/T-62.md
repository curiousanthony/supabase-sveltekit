The current Formation creation wizard (`src/routes/(app)/formations/creer/`) collects ~22 fields across 4 steps (Bases / Programme / Personnes / Financement) — including modules, formateurIds, apprenantContactIds, typeFinancement, montantAccorde, evaluationMode, suiviAssiduite. This conflicts with the Quest engine (which exists *to* drive progressive completion) and will fight the new financing model from T-50/T-53 (the wizard collects fields about to be deprecated: `typeFinancement`, `montantAccorde`, `financementAccorde`).

This ticket does NOT implement the new wizard. It is a **discovery ticket** that, when picked up, runs the full design council (product-analyst → ux-designer → human validation) per the team-orchestrator pattern, and emits a real implementation plan + follow-up tickets (T-63+).

**Why blocked by T-50..T-53:** the new wizard's financing decisions depend on the new `formation_funding_sources` model. Designing a wizard that collects today's `typeFinancement` enum would require re-doing it after T-53 ships. Sequencing matters.

**UX intention to validate (set by stakeholder):**

- Hand-holding momentum, not data collection. Marie commits to the *idea* of a formation in <30 seconds.
- The Quest engine owns the rest of the lifecycle.
- Likely minimum field set (to be validated by analyst):
  - Nom de la formation
  - Type (Intra / Inter / Sur-mesure) + CPF-éligible Switch
  - Client (entreprise) — optional
  - Dates approximatives — optional
  - Source de programme (vide / depuis bibliothèque) — optional
- Wizard ends with: *"Formation créée. N actions à compléter pour la rendre conforme."* — Zeigarnik open loop pointing at the quest list.
- Drop from wizard: modules, formateurs, apprenants, financement, évaluation, accessibilité, prérequis, public visé — these become quests.

## acceptance

- Product-analyst delivers `docs/team-artifacts/analysis/{date}-formation-wizard-audit.md` covering: Qualiopi minimums for "formation exists" status, current wizard field-by-field necessity audit, fields that must be in wizard vs. fields that can be quest-driven, alignment with new `formation_funding_sources` model
- UX designer delivers `docs/team-artifacts/design/{date}-formation-wizard-ux-review.md` + `docs/plans/{date}-T-formation-wizard-revamp.plan.md` covering: minimal field set, step structure (likely 1 step or 2 max), hand-off UX to Quests, mobile pattern, error recovery
- Human (Anthony) validates the analyst + designer outputs before any code is written
- Follow-up tickets created (T-63+) with concrete acceptance criteria
- This ticket then gets `status: done` with `decision: { hand_off: T-63, T-64, ... }`
- Pattern note: the existing wizard remains shipped and untouched until the new one is feature-complete; switchover happens via route swap, not progressive in-place edits
