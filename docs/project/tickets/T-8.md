Logos are stored as-uploaded in `workspace-logos`, which can grow storage quickly and leaves PDFKit and some email clients with unsupported types (WebP/SVG). Prefer **processing at upload time**: resize/compress, normalize to a safe format (e.g. PNG or JPEG within agreed max dimensions), and persist only the optimized object. PDF generation and Postmark footer images (`workspaceLogoUrl`) then use a consistent, small asset.

**Implementation gate — Plan Mode (mandatory):** Before writing implementation code, use **Cursor Plan Mode** to produce a detailed plan (constraints, library choice e.g. Sharp vs alternatives, max dimensions/file size, bucket MIME policy, migration for existing objects, failure modes). Record the approved plan path in frontmatter `plan` when ready; implement only after that plan exists.

## acceptance
- On logo upload: image is resized and/or compressed within defined limits; stored file format is PDFKit- and email-client-safe (PNG and/or JPEG as per plan)
- Supabase bucket policy and server validation align with allowed types after processing
- Settings UI copy recommends preferred extensions and approximate max dimensions/file size (and reflects what the server enforces)
- Generated PDFs and template emails show the workspace logo when one is configured (same stored asset as source of truth)
- Existing workspaces with legacy formats are addressed per plan (re-process on next upload, migration job, or documented manual step)
