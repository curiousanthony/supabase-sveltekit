Qualiopi indicateur 26 (référent handicap, dispositions d'accessibilité) is mandatory for every certified OF and currently has no field anywhere in the data model. Most OFs have a single référent for the whole organisation — so we store a workspace-level default and allow per-formation override only when needed (rare case: subcontracted formateurs with their own provisions).

## acceptance

- Schema additions: `workspaces.default_referent_handicap` (text), `workspaces.default_dispositions_handicap` (textarea/text), `formations.referent_handicap` (text, nullable), `formations.dispositions_handicap` (text, nullable)
- New section in `/parametres/organisme` for the workspace defaults
- Fiche "Logistique" gets a collapsed "Accessibilité" sub-card showing the workspace default chip ; toggle "Dispositions spécifiques pour cette formation" reveals the per-formation override fields (which fall back to the default when null)
- Convention and programme document templates updated to read `formation.referent_handicap ?? workspace.default_referent_handicap` (same for dispositions)
- All copy in French ; tooltip explains "Conformément à l'indicateur 26 du référentiel Qualiopi"

## log

### 2026-04-22 — Implementation

- **Schema** (`supabase/migrations/20260421160000_t56_accessibility_fields.sql`): Added `workspaces.default_referent_handicap`, `workspaces.default_dispositions_handicap`, `formations.referent_handicap`, `formations.dispositions_handicap` (all text, nullable). Drizzle schema updated in `src/lib/db/schema/{workspaces,formations}.ts`.
- **Workspace settings** (`/parametres/workspace`): New "Accessibilité (Qualiopi indicateur 26)" card with input + textarea, persisted via existing `saveSettings` action. Help text explains indicateur 26 and gives placeholder examples.
- **Fiche** (`/formations/[id]/fiche`): New `accessibility-section.svelte` component placed inside the Logistique card. Collapsed by default with one-line summary (effective referent + "dispositions standard"/"spécifiques"). When expanded: shows resolved values (formation override → workspace default), explicit "Modifier les dispositions spécifiques" button reveals a Switch-gated form. Yellow warning banner appears when nothing is configured at either level. `referentHandicap` and `dispositionsHandicap` added to `allowedFields` in `+page.server.ts`. Workspace defaults loaded once via `db.query.workspaces.findFirst` and exposed through `data.workspaceDefaults`.
- **PDF — Convention**: New "Article — Accessibilité aux personnes en situation de handicap" inserted between Moyens pédagogiques and Conditions financières. Renders `Référent handicap : ...` and `Dispositions d'accueil et d'accompagnement : ...` if either is present, else falls back to a generic statement. `FormationData.referentHandicap` / `dispositionsHandicap` are pre-resolved in `document-generator.ts` (`formation.x ?? workspace.defaultX`) so templates stay decoupled from the inheritance rule. `WorkspaceIdentity` also exposes the raw defaults for templates that may want to render them differently later.
- **Programme template**: Not updated — `programme` doc type isn't implemented yet in `document-generator.ts` (no case in the switch). Schema fields are ready to be consumed when that template ships.
- **Verification**: `bun run check` shows the same 72 errors / 36 warnings as baseline (none introduced by T-56). Smoke: `GET /formations/<id>/fiche` and `GET /parametres/workspace` both return 303 (auth redirect) → modules load without 500.

### Notes

- The plan referenced `/parametres/organisme` but the actual route is `/parametres/workspace` — used the existing route.
- One referent per workspace is the right default for the vast majority of OFs (single legal entity = single Qualiopi cert = single referent). The per-formation override exists for the rare subcontracted-formateur case mentioned in the plan, but is intentionally hidden behind a toggle to avoid overwhelming users with a field they'll never need.
