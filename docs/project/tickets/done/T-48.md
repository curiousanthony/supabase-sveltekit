When a Formation is created via the standard flow (`/formations/creer`), `getQuestsForFormation(...)` + `calculateDueDates(...)` seed `formation_actions` (Quests) and `quest_sub_actions` so the Suivi tab is populated. The `closeAndCreateFormation` action in `src/routes/(app)/deals/[id]/+page.server.ts` only inserts the `formations` row — no quests, no sub-actions, no due dates — so the resulting Suivi is empty and the formation is unusable until quests are added by hand.

Same seeding logic must run inside the Deal→Formation transaction. Quest filtering needs the formation `type` and `typeFinancement`; today the deal flow does not pass these (the Formation insert leaves both null), so the seeding call would currently include only quests with no `applicableTo` filter. Resolve by either (a) carrying `type`/`typeFinancement` over from the Deal where available (deal `fundingType`, deal `dealFormat` / programme metadata) before computing quests, or (b) seeding the unfiltered base set and letting the user complete the formation profile to refine. Pick the right posture during build.

Related but distinct: T-21 (devis carry-over + auto-complete the devis quest). T-48 unblocks T-21 — the devis quest must exist before it can be auto-completed.

## acceptance

- Creating a Formation from a Deal via `closeAndCreateFormation` seeds `formation_actions` (with first-quest `En cours`, others `Pas commencé`, `blockedByActionId` wired) and `quest_sub_actions` identically to `/formations/creer`
- Due dates are computed via `calculateDueDates` using whatever `dateDebut`/`dateFin` are known at creation time (null is acceptable — quest dueDate is nullable)
- Quest filtering uses the Deal's known `fundingType` / type signals when available; document the chosen posture in the ticket log
- Seeding happens inside the same transaction as the Formation insert; failure rolls everything back (Deal stays open, no partial Formation)
- Unit test covers: deal-with-funding → expected quest set seeded; deal-without-funding → base quest set seeded; transaction rollback on quest-insert failure leaves Deal `formationId` null
