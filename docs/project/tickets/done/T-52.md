Server-side counterpart to the new Finances UI. The Finances loader must return the multi-source funding lines plus the derived totals the synthesis card displays, and provide CRUD actions for funding sources. Invoice mutations gain an optional FK link to a funding source. All derivation logic is extracted to a shared service so the formation-header chip (T-57) can reuse it.

Per V1 decision: derived totals must include the **per-payer reste à charge split** (apprenant / entreprise / OF) so the UI can render distinct tiles.

## acceptance

- `src/routes/(app)/formations/[id]/finances/+page.server.ts` `load` returns `fundingSources: FundingSource[]` plus computed totals: `{ totalRequested, totalGranted, resteACharge: { total, byPayer: { apprenant, entreprise, OF } }, percentCovered, statutGlobal }`
- New actions: `createFundingSource`, `updateFundingSource`, `deleteFundingSource` — all enforce workspace ownership, all write to `formation_audit_log` with action types `funding_source_created`/`funding_source_updated`/`funding_source_deleted`
- `createInvoice` and `updateInvoice` accept optional `fundingSourceId` form field, validated as UUID belonging to the same formation (404 otherwise)
- Server-side derivation extracted to `src/lib/services/funding-summary.ts` exposing `getFundingSummary(fundingSources, prixConvenu)` — pure function, fully unit-testable
- Per-payer split computed deterministically: a funding source's unfunded portion (`requestedAmount - grantedAmount`, or `prixConvenu` share when `granted = 0`) is attributed to the source's `payer_type`
- Unit tests in `src/lib/services/funding-summary.test.ts` cover: zero sources, single source fully funded, multi-source partial, refused source, mixed payer types
