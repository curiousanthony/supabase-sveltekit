# Learnings

One-line insights discovered during development. Reviewed by team-architect every 10 entries.

- 2026-04-22 T-56: When a feature has a "default at level N, override at level N+1" shape (workspace default → per-formation override for Qualiopi indicator 26), resolve the fallback **once at the loader/service boundary** (`formation.referentHandicap = formation.referentHandicap ?? workspace.defaultReferentHandicap`) and let templates/UI components consume the already-resolved value. Pushing the `??` into every PDF builder, every component, every export turns a one-line invariant into a leaky abstraction that will silently diverge the day someone forgets the fallback. The `WorkspaceIdentity` interface can still expose the raw default for future templates that need to render "valeur par défaut" labels — but the formation's resolved value is what 95% of consumers want. Plan said `/parametres/organisme` but actual route is `/parametres/workspace` — always grep for the route before trusting plan paths; planning docs drift faster than code.
- 2026-04-21 T-54: A "is this feature relevant?" Switch backed by *defaults* must never destroy the underlying field values when toggled off — wire it as a writable override (`let override = $state<boolean | null>(null); const isCertifying = $derived(override ?? hasAnyValue)`) so an accidental click is recoverable and the UI stays consistent with persisted data on refresh. Ticket dependencies in the project tracker should reflect *causality of work*, not "logical follow-up". T-54 was marked `blocked_by: T-51` because T-51 drops the `tjm_formateur` column the Fiche references — but T-54 only removes the UI references, which is a hard prerequisite *for* T-51 (you can't drop a column the UI still writes to). Reversed dependency caused T-54 to sit "blocked" in the board while it was actually ready; always ask "what data/code must exist before I can write this code?", not "what feels like the natural next step?". For long-form text inputs (>1 line) in inline-edit pages, the Modifier/Enregistrer pattern beats blur-save: Marie types a paragraph for `publicVise`, gets a Slack ping, clicks away → blur-save would fire mid-thought; explicit Enregistrer with ⌘+Entrée shortcut + Annuler gives her control without breaking flow.
- 2026-04-21 T-53: When `$derived.by(...)` returns different shapes on different branches (e.g. early-return `[]` vs a `.filter().sort()` chain on tuples), svelte-check infers the broad union and every field access downstream fails. Anchor the inference explicitly with a type parameter (`$derived.by<Array<{...}>>(() => ...)`) AND always `.map(([k,v]) => ({...}))` before returning — resist the urge to return tuples; named-field objects survive HMR and read well in templates. Also: when porting legacy Card-bound fields (`formation.montantAccorde`) to a derived value (`summary.totalGranted`), re-anchor *every* dependent calculation (Marge) in one pass — leaving a single orphan reference keeps the deprecated field alive through transitive UI, which defeats the "deprecate in one release cycle" plan.
- 2026-04-21 T-53: Smart defaults in forms must be *additive*, never destructive — only populate a field when the user's value is empty. The rule "source changes → fill payer_label" sounds right until Marie types "Mon OPCO perso" and then reselects the source to double-check: her input vanishes and trust erodes. Keep a tight guard (`if (!fsPayerLabel)`) so the dialog is helpful on first pass and safe on edit. For the shared label lookup, a pure `src/lib/utils/*.ts` module (no Svelte, no DB) that exports descriptors keyed by enum value is the right seam — both the Finances tab and the forthcoming header chip (T-57) can import it without introducing a runtime cost or an import cycle.
- 2026-04-21 T-52: When extracting derivation logic to a pure service for "unit-testable" acceptance, stub the Drizzle schema in the test with `vi.mock('$lib/db/schema', () => ({ formationFundingSources: Symbol(...) }))` — the service only imports the table object as a *type anchor* for `InferSelectModel`, so a symbol satisfies the module graph without dragging postgres-js/pglite into vitest. Anchor the function on raw row shapes the DB returns (numeric columns come back as strings, null → 0 via a small `toNumber` helper) rather than on a domain model that requires a mapper — keeps the loader a one-liner (`getFundingSummary(rows, prixConvenu)`) and the service trivially testable. Unallocated reste-à-charge (gap between formation price and per-source claims) defaults to `apprenant` rather than being hidden, because Marie needs to *see* the gap; `percentCovered` stays uncapped so over-funding surfaces as > 100%.
- 2026-04-21 T-55: Bounded integer columns need both a DB CHECK (`BETWEEN 1 AND 8`) AND server-side coercion that rejects out-of-range values to null; relying on the DB alone surfaces ugly 23514 errors to the user, while relying on the server alone lets a service-role caller bypass validation. Mirror the bound in both layers; do it silently (out-of-range → null) rather than throwing so existing inline-edit UX stays smooth.
- 2026-04-21 T-63: Drizzle handles bidirectional schema imports (formations.ts ↔ funding-sources.ts) cleanly because `foreignKey({ columns: [t.x], foreignColumns: [otherTable.id] })` runs inside the lazy `(table) => [...]` callback; the import statement returns the partially-initialized module, but the callback fires after both modules finish loading. Verify with `bun -e "import('./schema').then(s => s.tableA.fkColumn)"` before relying on it. Don't try to "fix" the cycle by moving the FK into a single file — the canonical pattern in this codebase is one table per file, and the cycle is harmless.
- 2026-04-21 T-63: Project-tracker hygiene — when creating a batch of related tickets from a planning doc, the plan's `ticket_to_task_map` and the actual ticket files can drift if a number is grabbed by an unrelated ticket created in parallel. Always cross-check `ls docs/project/tickets/T-*.md` against the plan map before relying on the dependency graph; mismatches manifest as `blocked_by` pointing at the wrong ticket and silently stall the chain. Also: `## id: T-N` as a markdown heading inside YAML frontmatter breaks `scripts/board.sh` silently (the script's `field()` function fails under `set -e` + `pipefail` when grep finds no `^id:`). Stick to bare `id: T-N` between the `---` markers as `project-tracker.mdc` mandates.
- 2026-04-21 T-63: For idempotent Supabase enum/FK/trigger creation, wrap each in a `DO $$ BEGIN IF NOT EXISTS … END $$` block (checking `pg_type` for enums, `pg_constraint` for FKs, `pg_trigger` for triggers). `IF NOT EXISTS` works directly only for `CREATE TABLE`/`COLUMN`/`INDEX` — not for `CREATE TYPE`, `ADD CONSTRAINT`, or `CREATE TRIGGER`. The codebase's `updated_at` trigger function is `public.set_updated_at()`, not `moddatetime()` — use the existing helper to stay consistent.
- 2026-04-21 T-20: When extending a polymorphic-FK RLS predicate (one row scoped via either of two parent tables), use **AND across `IS NULL OR EXISTS(parent in my workspace)`**, never `OR` between two `EXISTS`. The OR form lets a member of workspace A insert/update a row whose A-side FK is theirs but whose B-side FK points at a victim's parent — only one branch needs to pass and the row becomes visible/writable to workspace B via the deal-side SELECT branch (stored content + phishing surface). The AND form requires every set FK to resolve to the caller's workspace, fails closed on legacy/buggy cross-tenant rows, and pairs naturally with a CHECK that at least one FK is set. Always also add explicit `WITH CHECK` to UPDATE policies — relying on PG's USING fall-through has historically been a foot-gun for pivot attacks.
- 2026-04-10 T-8: Replace workspace storage objects only after a successful new upload + DB update; validate object keys against `workspaceId/` before `remove()`; use Sharp `limitInputPixels` against decompression DoS
- 2026-04-10 T-39: Match an existing route’s Kanban (`flex` + `overflow-x-auto`, `min-w`/`max-w` columns) instead of inventing a new grid breakpoint pattern
- 2026-04-09 T-1: French PDF amounts need Intl currency + replace U+202F/U+00A0 so Helvetica does not garble thousands separators
- 2026-04-09 T-2: Centralize PDF_TIMEZONE in shared for every template date/time helper
- 2026-04-09 T-3: `default` + `assertNever` on DocumentType switch catches missing cases at compile time
- 2026-04-09 T-4: Typed wrapper for pdfmake CJS singleton avoids `any` on createRequire
- 2026-04-09 T-5: Certificat attendance — innerJoin seances on formationId avoids loading all contact émargements
- 2026-04-09 T-32: Hand-written SQL migrations can get out of sync with Supabase tracking — always verify schema_migrations table before `migration up`

- 2026-04-10 T-7: Storage policies use (storage.foldername(name))[1]::uuid to extract entity ID from path convention {entityId}/filename.ext
- 2026-04-10 T-28: When raw-HTML email path exists alongside template-based path, remove raw HTML — all Postmark sends should use templates for consistency and deliverability

- 2026-04-10 T-35: When linking quest sub-actions to document lifecycle, use sub-action orderIndex (not title matching) to identify specific actions within a quest
- 2026-04-10 T-36: Compliance warnings belong in the layout load (not per-page) so both Documents and Suivi share the same computed warnings from a single lightweight query
- 2026-04-13 T-12: Pre-validate lifecycle transitions before generating a replacement doc — generateDocument creates storage + DB rows that become orphans if replaceDocument rejects afterward
- 2026-04-13 T-11: Svelte 5 {#snippet} avoids template duplication when the same card layout is rendered in flat, grouped, and nested contexts — define once, @render everywhere
- 2026-04-13 T-17: Vercel Cron Jobs are the natural fit for SvelteKit-on-Vercel scheduled tasks — configure in vercel.json crons array, auth via CRON_SECRET bearer token, and use service-role Supabase client (not user session) since cron runs without auth context
- 2026-04-20 T-13: Server-side preflight must run on every generation entry point (e.g. Documents actions and Suivi quest generation); skipping one path is a compliance gap even if the UI looks complete.
- 2026-04-20 T-14: For batch async work that may partial-fail, share one `batchId` across every audit-log entry (success and failure) so post-hoc replay/grouping is trivial — and pass the `tx`/`db` client to `logAuditEvent` so the throwing branch actually fires the rollback (silent fire-and-forget hides the recovery path).
- 2026-04-21 T-48: When two creation paths produce the same entity (e.g. Formation via /formations/creer vs deal closeAndCreateFormation), seeding-related side-effects (quests, sub-actions, default rows) MUST be replicated on both paths or extracted into a shared helper from day one — silent divergence between paths produces "ghost-empty" entities (e.g. Suivi tabs with zero quests) that look like UI bugs but are actually missing seed data.
- 2026-04-21 T-46: For "the data must come from a trusted source" invariants, a TypeScript brand type (`type Authenticated<X> = X & { readonly __brand: 'Authenticated' }` + a single mint helper) is more reliable than docstrings — the type system forces every caller to acknowledge the boundary at the call site, and refactors that introduce a new caller break the build instead of silently passing client input through. Pair with an RLS WITH CHECK clause (`user_id = auth.uid()`) for defense in depth so the SQL layer also rejects spoofed inserts even if Drizzle's RLS-bypass posture changes.
