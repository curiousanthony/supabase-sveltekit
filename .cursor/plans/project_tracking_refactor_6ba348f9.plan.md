# Project Tracking Refactor for Agent Efficiency

## Problem

Agents read ~2,500 tokens across 4 files just to find the next work item. There are no ticket IDs, no agent assignment, no plan-to-ticket linking, and the orchestrator has no concept of "what to work on next." Agents can also silently ignore soft instructions ("only read the frontmatter") leading to token waste and drift.

## Architecture Principles

These principles govern every design choice in this system:

1. **Mechanical constraints > instructional constraints.** Don't tell agents "only read X" -- give them a script that physically returns only X. Don't tell agents "keep board in sync" -- derive the board from ticket files via a script. Deterministic scripts eliminate AI interpretation errors.

2. **Progressive disclosure.** Information is layered by cost. Each layer is only read when needed:
   - Layer 0: `board.md` (~50 tokens) -- "what exists?"
   - Layer 1: `tickets/T-N.md` (~100 tokens) -- "what is this ticket?"
   - Layer 2: Plan frontmatter (~50 tokens, extracted by script) -- "what's the progress?"
   - Layer 3: Full plan body (~500 tokens) -- "what's the implementation approach?"
   - Layer 4: Decision doc (~1,000-4,000 tokens) -- "what were the design decisions?"
     Agents must not jump layers. The orchestrator reads Layer 0-1 at pickup. Layers 2+ are read only during the relevant phase.

3. **Single source of truth.** Every fact lives in exactly one place. Ticket YAML is authoritative for status, priority, and all ticket fields. The board is a **derived artifact** regenerated by script -- never edited manually. No sync, no drift.

4. **Lean orchestrator, rich skill.** The orchestrator rule is `alwaysApply: true` (~1,140 tokens loaded into every conversation). Detailed ticket operations live in the `project-tracker` skill (loaded on-demand). The orchestrator adds ~15 lines of references, not ~60 lines of logic.

## Design Decisions

Validated through 4 rounds of questions:

1. **Format**: Machine-first, minimal tokens. YAML frontmatter for structured fields, minimal Markdown body.
2. **File structure**: Two-tier -- derived board index + individual ticket files.
3. **Plan location**: Cursor-native (`~/.cursor/plans/` for Cursor 3, `.cursor/plans/` for legacy). Ticket `plan` field stores the filename slug (location-agnostic).
4. **Autonomous scope**: Orchestrator auto-picks highest-priority ticket, pauses with one-line confirmation, then runs autonomously with pauses only at existing phase gates.
5. **Migration**: Full migration of backlog items. Shipped items get `T-S` prefix IDs and go to `done/`. Selective plan linking.
6. **Agent writes**: Orchestrator runs `scripts/board.sh` after ticket changes. Subagents append to ticket `## log`. Project-manager handles ticket creation/archival/changelog.
7. **Team artifacts**: Keep `docs/team-artifacts/`, link via ticket IDs in filenames, orchestrator passes specific artifact paths to subagents. English language, preserving French user-facing terms.

## board.md (derived artifact, ~50 tokens)

Generated by `scripts/board.sh`. Agents read this; agents never write to it directly. After any ticket status change, the agent runs `scripts/board.sh` to regenerate.

```markdown
## sprint

T-1 P1 Fix price formatting on documents
T-2 P1 Add timezone to date formatting
T-3 P2 Exhaustive switch for DocumentType
T-4 P2 Type pdfmake import
T-5 P2 Optimize certificat emargements query
```

Sorted by priority within each status group. No legends, no tables, no separators. If board exceeds 40 lines, P3 backlog items are excluded (they exist in ticket files but not on the board).

### scripts/board.sh

```bash
#!/usr/bin/env bash
set -euo pipefail
root="$(git rev-parse --show-toplevel)"
dir="$root/docs/project/tickets"
out="$root/docs/project/board.md"
learnings="$root/docs/project/learnings.md"

# --- Robust YAML field extraction ---
# Uses grep + sed instead of awk -F': ' to handle colons in values (e.g. "title: Fix: something")
field() {
  # $1 = field name, $2 = file path
  grep "^${1}:" "$2" | head -1 | sed "s/^${1}: *//"
}

emit() {
  local status="$1"
  for f in "$dir"/T-*.md; do
    [ -f "$f" ] || continue
    local s
    s=$(field status "$f")
    [ "$s" = "$status" ] || continue
    local id p t
    id=$(field id "$f")
    p=$(field priority "$f")
    t=$(field title "$f")
    printf '%s %s %s\n' "$id" "$p" "$t"
  done | sort -t' ' -k2,2
}

# --- Generate board ---
{ echo "# Board"
  echo ""
  echo "## sprint"
  emit sprint
  echo ""
  echo "## backlog"
  emit backlog
} > "$out"

echo "Board regenerated: $(grep -c '^T-' "$out") tickets."

# --- Orphan detection: tickets stuck in in_progress ---
orphans=""
for f in "$dir"/T-*.md; do
  [ -f "$f" ] || continue
  s=$(field status "$f")
  if [ "$s" = "in_progress" ]; then
    orphans="$orphans $(field id "$f")"
  fi
done
if [ -n "$orphans" ]; then
  echo "WARNING: Orphaned in_progress tickets:$orphans — resume or reset these."
fi

# --- Learnings review nudge ---
if [ -f "$learnings" ]; then
  total=$(grep -c '^- ' "$learnings" 2>/dev/null || echo 0)
  reviewed=$(grep -c '^## Reviewed' "$learnings" 2>/dev/null || echo 0)
  pending=$((total - reviewed * 10))
  if [ "$pending" -ge 10 ]; then
    echo "ACTION: $pending unreviewed learnings. Run team-architect review."
  fi
fi
```

**Why grep+sed instead of awk -F': '**: A title like `Fix: price formatting` contains a colon. `awk -F': '` would split on the first `: ` and discard everything after the second colon. `grep + sed` extracts the full line and strips only the field prefix, preserving the entire value.

**Orphan detection**: Prints a WARNING if any tickets are stuck in `in_progress` status. The orchestrator sees this output and should offer to resume or reset them in Phase 0.

**Learnings nudge**: Counts unreviewed entries in `learnings.md`. After 10+ entries since the last `## Reviewed` marker, prints an ACTION message. The orchestrator sees this and should suggest a team-architect review.

## Ticket File Format

Example: `docs/project/tickets/T-1.md`

```yaml
---
id: T-1
title: Fix price formatting on documents
status: sprint
priority: P1
type: bug
estimate: small
chunk: 1
blocked_by: []
plan: null
decision: "2026-04-07-document-generation-system"
created: 2026-04-08
assignee: null
artifacts: []
---
Amounts display "2/ 600€" instead of "2 600,00 €" in devis and convention pricing.

## acceptance
- Prices format as "2 600,00 €" on all generated PDFs
- Verified on devis, convention, and all price-displaying templates

## log
```

### Fields

- `status`: `sprint`, `backlog`, `in_progress`, `done`
- `type`: `bug`, `feature`, `tech-debt`, `security`, `docs`
- `estimate`: `small` (1 session), `medium` (2-3 sessions), `large` (full feature cycle). Set during brainstorm. Helps orchestrator decide batching.
- `blocked_by`: list of ticket IDs (e.g., `[T-1, T-2]`). Orchestrator skips blocked tickets during pickup.
- `plan`: filename slug (e.g., `chunk_1_implementation_4fd2722e`). Location-agnostic.
- `decision`: slug of decision doc in `docs/decisions/`. Metadata for traceability -- NOT read at pickup (only read during Phases 1-2).
- `assignee`: set by orchestrator at pickup (e.g., `"session-2026-04-08"`).
- `artifacts`: paths to team-artifact files for this ticket.
- `## acceptance`: top 5-7 criteria inline. For complex features, reference a spec file.
- `## log`: append-only. `- {date} {agent}: {summary}`

## project-tracker Skill (the operations hub)

Rewrite [`.cursor/skills/project-tracker/SKILL.md`](.cursor/skills/project-tracker/SKILL.md) as the central operations hub. This is where all detailed, deterministic ticket logic lives. Agents invoke this skill for any ticket operation.

### Skill structure

```markdown
# Project Tracker

## File Structure

- `docs/project/board.md` -- DERIVED, regenerated by `scripts/board.sh`. Never edit manually.
- `docs/project/tickets/T-{N}.md` -- Source of truth for each ticket.
- `docs/project/tickets/done/` -- Completed tickets moved here.
- `docs/project/changelog.md` -- Append-only shipped history.
- `docs/project/learnings.md` -- Append-only session insights. Reviewed by team-architect.
- `docs/project/roadmap.md` -- High-level direction, ~15 lines.

## Progressive Disclosure (MANDATORY)

- Layer 0: board.md (~50 tok) -- always read first
- Layer 1: ticket file (~100 tok) -- read when picking up work
- Layer 2: plan frontmatter (~50 tok) -- check progress (use script below)
- Layer 3: full plan (~500 tok) -- only during implementation phases
- Layer 4: decision doc (~1-4K tok) -- only during brainstorm/design phases
  Never jump layers. Each layer is gated by the previous.

## Operations

### Read plan progress (deterministic)

To check plan status without reading the full body, run:
awk '/^---$/{c++; next} c==1' <PLAN_FILE>
This returns only the YAML frontmatter (todos with statuses). Do NOT use Read on the full file.

### Regenerate board (deterministic)

After ANY ticket status/priority change, run:
bash scripts/board.sh
Check its stdout for:

- "WARNING: Orphaned in_progress tickets" -- resume or reset before continuing
- "ACTION: N unreviewed learnings" -- suggest team-architect review
  Never edit board.md manually. It is a derived artifact.

### Create ticket

1. Find next ID:
   ls docs/project/tickets/T-\*.md | grep -oP 'T-\K\d+' | sort -n | tail -1
   Add 1 to get next ID.
2. Create docs/project/tickets/T-{N}.md using the YAML template below.
3. Run: bash scripts/board.sh

### Update ticket status

1. Edit the `status:` line in the ticket's YAML frontmatter.
2. Run: bash scripts/board.sh

### Close ticket (5 required steps)

All 5 steps are required. The learning parameter is NOT optional.

1. Set `status: done` in ticket YAML.
2. Move file: mv docs/project/tickets/T-{N}.md docs/project/tickets/done/
3. Run: bash scripts/board.sh
4. Append one line to `docs/project/changelog.md` under today's date heading:
   - T-{N} {one-line summary}
5. Append to `docs/project/learnings.md`:
   - {date} T-{N}: {learning}
     The learning must be a concrete, reusable insight. If genuinely nothing was
     learned, write: `- {date} T-{N}: none`

### Board cap

If board.md exceeds 40 lines after regeneration, the script excludes P3 backlog items.
They still exist as ticket files and can be found with:
ls docs/project/tickets/T-\*.md
```

### Why this works

- **Deterministic scripts** for board regeneration and plan frontmatter reading eliminate AI interpretation errors
- **Required parameters** (learning in Close Ticket) prevent steps from being skipped
- **Progressive disclosure rules** prevent agents from reading 4,000-token decision docs at startup
- **Operations as recipes** -- agents follow numbered steps, not vague instructions
- **Script output drives behavior** -- orphan warnings and learning nudges are mechanical, not remembered
- **Skill is on-demand** -- only loaded when ticket operations are needed (not every conversation)

## Self-Improvement Loop (enforced)

### How it works

```mermaid
graph TD
    close["Phase 7b: Close Ticket"] -->|"required param"| learning["Append learning to learnings.md"]
    learning --> boardSh["scripts/board.sh runs"]
    boardSh -->|"counts unreviewed"| nudge{"10+ pending?"}
    nudge -->|"yes"| action["Prints ACTION: run team-architect review"]
    nudge -->|"no"| silent["No message"]
    action --> orchestrator["Orchestrator sees it in Phase 0"]
    orchestrator --> teamArch["Launches team-architect"]
    teamArch -->|"safe"| autoApply["Auto-apply: append to skill, add to learnings"]
    teamArch -->|"moderate"| flagApply["Apply + flag to user"]
    teamArch -->|"risky"| proposeOnly["Propose only, require user approval"]
    teamArch --> reviewMarker["Appends '## Reviewed YYYY-MM-DD' to learnings.md"]
    reviewMarker --> boardSh
```

### Enforcement mechanisms

**1. Learning capture is a required parameter in Close Ticket.**

The project-tracker skill's Close Ticket recipe requires a `learning` argument. The agent must provide either a concrete insight or the literal string `"none"`. This makes it a required step, not an optional afterthought:

```
Close ticket recipe:
1. Set status: done
2. Move to done/
3. Regenerate board: bash scripts/board.sh
4. Append to changelog.md
5. Append to learnings.md: "- {date} T-{id}: {learning}"    <-- REQUIRED, even if "none"
```

**2. Review trigger is mechanical, not remembered.**

`scripts/board.sh` counts unreviewed learnings every time it runs. When the count exceeds 10, it prints `ACTION: N unreviewed learnings. Run team-architect review.` The orchestrator sees this stdout in Phase 0 (board.sh runs during orphan check) and should act on it.

**3. Team-architect applies improvements with tiered safety.**

When the team-architect reviews learnings, it classifies each proposed improvement:

- **Safe** (examples: add a note to a skill, add a test case pattern, update a checklist): auto-apply the change and record it in `docs/team-changelog.md`.
- **Moderate** (examples: update an agent definition, add a step to a recipe): apply the change, flag it to the user in the session summary. User can revert.
- **Risky** (examples: change the orchestrator rule, modify phase gates, alter skip conditions): propose only. Write the proposal to `docs/team-artifacts/management/` and present to the user for approval. Do not apply.

**4. Review marker resets the counter.**

After reviewing, the team-architect appends `## Reviewed YYYY-MM-DD` to `learnings.md`. The board.sh script counts entries after the last `## Reviewed` marker, so the counter resets. The cycle continues.

### What a good learning looks like

The project-tracker skill includes examples to guide learning quality:

```
GOOD (actionable, reusable):
- 2026-04-08 T-1: French currency formatting requires toLocaleString('fr-FR') not toFixed(2)
- 2026-04-08 T-5: Drizzle joins on formation_apprenants need explicit select to avoid N+1
- 2026-04-08 T-S2: pdfmake CJS via createRequire needs setFonts() before createPdf()

BAD (vague, not reusable):
- Fixed the bug
- Updated the formatting
- Made it work
```

### learnings.md format


## Risk Mitigations

Every identified risk has a concrete, mechanical mitigation built into the system:

### Risk 1: Agent edits board.md directly (bypassing single source of truth)

**Mitigation**: The orchestrator rule and project-tracker skill both state "board.md is derived -- never edit manually." But this is instructional. The mechanical safeguard: `board.sh` overwrites board.md entirely on every run. Even if an agent edits it, the next `board.sh` invocation restores correctness. The damage window is one operation.

### Risk 2: YAML parsing breaks on special characters

**Mitigation**: `board.sh` uses `grep + sed` instead of `awk -F': '`. This handles colons, quotes, and brackets in values. The `test-board-script` todo specifically tests edge cases:

- Title with colon: `title: Fix: price formatting`
- Title with quotes: `title: Add "Hérité du deal" badge`
- Empty blocked_by: `blocked_by: []`
- Missing optional fields
- Non-ASCII characters (French accents)

### Risk 3: Orphaned tickets stuck in `in_progress`

**Mitigation**: `board.sh` detects and prints orphaned tickets in its output. Phase 0 in the orchestrator reads board.sh output (via the project-tracker skill's board check) and offers to resume or reset orphaned tickets before picking up new work. The skill's Phase 0 recipe:

```
1. Run: bash scripts/board.sh (regenerate + checks)
2. If WARNING about orphans: read orphaned ticket(s), ask user "Resume T-{id} or reset to sprint?"
3. If ACTION about learnings: suggest team-architect review before picking up work
4. Read board.md, select next ticket
```

### Risk 4: Learning quality is poor

**Mitigation**: The project-tracker skill includes good/bad examples (see above). The Close Ticket recipe rejects empty learnings -- the agent must provide the string `"none"` explicitly, which is a conscious decision rather than forgetting. The team-architect reviews learnings and can flag patterns of low-quality entries as a "moderate" improvement to the relevant agent's definition.

### Risk 5: Scale ceiling (100+ tickets slow board.sh)

**Mitigation**: At current scale (31 tickets), board.sh runs in <100ms. The script's P3 backlog exclusion (40-line cap) keeps the board compact as tickets grow. For future scale (100+), the migration path is:

1. **Now**: Shell scripts in the project-tracker skill (this plan)
2. **After 1 week of use**: Extract a lightweight CLI wrapper (`scripts/mm`) that combines common operations into single commands. Reduces token cost per operation from ~420 to ~50.
3. **If CLI proves valuable**: Wrap it as a custom MCP server for structured I/O and tool-descriptor auto-loading. Only justified at 100+ tickets or 5+ ticket operations per session.

This progression is documented so it isn't lost. Each stage validates the need for the next.

### Risk 6: Agent doesn't invoke the project-tracker skill

**Mitigation**: The orchestrator rule (alwaysApply) explicitly references the skill: "Use the `project-tracker` skill for all ticket operations." The orchestrator is the entry point for every non-trivial session. Since it's always loaded, the reference is always present. Additionally, the Phase 0 and 7b descriptions in the orchestrator both say "Use the project-tracker skill" -- agents encounter this instruction at the two most critical points (session start and end).

## Future Work (out of scope for this plan)

Extract common script operations into a unified CLI at `scripts/mm`:

Token savings: ~420 tokens per operation -> ~50 tokens. Justification threshold: if ticket operations happen 3+ times per session regularly.

### Phase E: MCP Server (only if CLI proves valuable)

Wrap the CLI as a custom MCP server. Benefits: structured JSON I/O, tool descriptors auto-loaded (agents don't need to read the skill), input validation, compound operations become atomic. Cost: server process management, MCP config, debugging complexity. Only justified at 100+ tickets or if structured I/O materially improves agent reliability.

### Skillsets for specialized agents

If specific agents develop recurring deterministic needs (e.g., security-analyst RLS audit script, team-architect learnings analysis), create dedicated skills with embedded scripts. Currently only the project-tracker skill warrants this level of tooling.
